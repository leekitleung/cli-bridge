# ADR-0006: Multi-provider AgentTeam

Status: ACCEPTED

Date: 2026-06-12
Acceptance: Senior review passed (2026-06-12). Accepted with conditions on the
            v2.4b implementation handoff (see "Acceptance Conditions" below).
            The v2.3 safety boundary is unchanged: sequential, concurrency 1,
            patch-only, read-only conflict reports. No implementation is
            authorized until a v2.4b execution handoff is created.

## Context

v2.3 implemented the AgentTeam Sequential MVP:

```text
TeamSpec create -> approve/cancel -> sequential slot advance ->
SlotArtifact record -> read-only conflict report -> slot lifecycle audit ->
Console Teams read-only visibility
```

The implemented v2.3 boundary is single-provider, sequential, patch-only, and
non-autonomous. Bridge-governed parallel slots, worktree isolation,
workspace-write auto-apply, commit, push, merge, scheduler behavior, and
provider-driven execution authority remain forbidden.

The post-v2.3 planning handoff identifies Track C as "Multi-provider AgentTeam
Planning", required before v2.4b implementation. The open design items are:

- provider identity and session identity;
- capability declaration parity across providers;
- provider failure and partial-result semantics;
- artifact normalization across provider outputs;
- review strategy for conflicting provider outputs.

This ADR decides whether v2.4b may introduce multi-provider AgentTeam support
inside the existing v2.3 safety boundary. It does NOT authorize implementation
while PROPOSED.

## Decision

### 1. Whether multi-provider AgentTeam is allowed

**Decision**: PERMIT, only for sequential, patch-only AgentTeam
orchestration.

v2.4b MAY allow a TeamSpec to bind different logical slots to different
approved providers, provided that:

- bridge-governed concurrency remains 1;
- execution remains sequential;
- isolation remains `patch-only`;
- all providers declare equivalent capability metadata;
- all provider outputs are normalized into the existing SlotArtifact shape;
- all conflicts are read-only review evidence and never auto-merged.

### 2. Scope

The first v2.4b multi-provider slice is limited to:

- Multi-provider TeamSpec shape:
  - existing team-level provider remains supported;
  - each logical slot MAY declare its own `providerId` and `endpointId`;
  - missing slot provider defaults to the team-level provider for backward
    compatibility.
- Provider identity model:
  - stable bridge provider id;
  - endpoint id;
  - provider kind (`codex`, `claude`, `model-api`, or future registered kind);
  - display label;
  - evidence metadata for claimed capabilities.
- Provider session identity model:
  - optional external session id per slot;
  - bridge-generated slot run id;
  - audit correlation across provider, team, slot, plan step, and artifact.
- Capability declaration parity:
  - all providers use the same capability keys and evidence-tier rules;
  - unknown capabilities fail closed.
- Artifact normalization:
  - all provider outputs become SlotArtifact records with normalized proposed
    files, summary, verification notes, redaction state, provider id, endpoint
    id, and session/run correlation.
- Conflict review:
  - conflict reports may compare artifacts from different providers;
  - conflicts remain read-only evidence and do not apply, merge, or choose a
    winner automatically.

### 3. Provider identity

Provider identity MUST be explicit. Tool names do not grant authority.

Minimum provider identity:

```text
ProviderIdentity {
  providerId: string;       // stable within bridge config/store
  endpointId: string;       // registered endpoint
  kind: string;             // codex | claude | model-api | ...
  label: string;            // display only
  evidenceTier: 1 | 2 | 3 | 4;
}
```

Rules:

- `providerId` and `endpointId` are both audit-visible.
- `label` is display-only and has no policy meaning.
- A provider cannot inherit authority from another provider.
- A model API role such as PlannerModel or CriticModel remains
  `canExecute=false` unless a separate ADR explicitly changes that role's
  authority. ADR-0004 and ADR-0005 continue to bind.

### 4. Provider session identity

Provider sessions are correlation evidence, not authority.

Minimum slot run identity:

```text
SlotRunIdentity {
  teamId: string;
  slotId: string;
  planStepId: string;
  providerId: string;
  endpointId: string;
  bridgeRunId: string;
  externalSessionId?: string;
}
```

Rules:

- `bridgeRunId` is generated by cli-bridge for each slot attempt.
- `externalSessionId` is optional and provider-supplied.
- Missing external session identity does not block sequential patch-only slot
  execution, but it MUST be recorded as unavailable.
- External session identity cannot authorize retries, dispatch, workspace-write,
  merge, or gate bypass.

### 5. Capability declaration parity

All providers MUST declare capabilities using the same schema. Unknown means
unavailable.

Minimum capability keys:

```text
ProviderCapability {
  canReview: boolean;
  canProposePatch: boolean;
  canVerify: boolean;
  canExecute: boolean;
  canUseModelApi: boolean;
  productNativeParallelism: 'confirmed' | 'reported' | 'unknown';
  productNativeParallelismEvidenceTier: 1 | 2 | 3 | 4;
  bridgeGovernedParallelSlots: false;
  maxConcurrentBridgeSlots: 1;
  isolationModes: ['patch-only'];
  supportedModes: string[];
}
```

Rules:

- `bridgeGovernedParallelSlots` remains `false` in this ADR.
- `maxConcurrentBridgeSlots` remains `1`.
- `isolationModes` is limited to `patch-only`.
- `canExecute=true` does not grant workspace-write without ADR-0003 plan and
  per-step gate requirements.
- `canUseModelApi=true` does not grant execution authority.
- WorkBuddy remains `canExecute=false`.

### 6. Failure and partial-result semantics

Multi-provider execution MUST fail closed and preserve sequential control.

Failure rules:

- If a slot provider is unavailable before slot start, the slot is not started
  and the team remains paused/failed according to the existing sequential
  lifecycle.
- If a provider fails during a slot, that slot records failure metadata and the
  team stops; later slots do not start automatically.
- If a provider returns a partial artifact, the artifact is recorded only if it
  passes redaction and SlotArtifact validation; otherwise the failure is
  recorded without a reusable artifact.
- A later provider cannot continue a failed provider's slot unless a separate
  human-approved retry/reassignment handoff authorizes that behavior.
- No provider failure triggers automatic replanning, model arbitration, slot
  reassignment, merge, commit, push, or workspace write.

### 7. Artifact normalization

All provider output used by AgentTeam must normalize to SlotArtifact-like
records before it can participate in conflict reporting or memory/verification
projection.

Minimum normalized fields:

- `teamId`, `slotId`, `planStepId`;
- `providerId`, `endpointId`, `bridgeRunId`, optional `externalSessionId`;
- `summary`;
- `proposedFiles`;
- `verificationNotes`;
- `outputRedacted`;
- `createdAt`.

Raw provider output remains subject to existing redaction rules. Unredacted raw
provider output MUST NOT be stored.

### 8. Conflict review strategy

Conflict review remains read-only.

Rules:

- Conflict detection compares normalized `proposedFiles` across all artifacts,
  including artifacts from different providers.
- Conflict reports identify provider ids and slot ids involved.
- Conflict reports do not pick winners.
- Conflict reports do not apply patches, merge files, edit workspace content,
  commit, push, or open PRs.
- Human review remains required for any future apply/merge workflow.

### 9. API and UI surface

Implementation SHOULD preserve existing TeamSpec routes and add fields rather
than creating a new product surface.

Possible API changes for an implementation handoff:

- Extend TeamSpec create payload to accept per-slot provider binding.
- Extend TeamSpec and artifact responses with provider/session correlation.
- Extend conflict report response with provider ids for conflicting artifacts.
- Extend console Teams read-only view to show provider per slot.

No UI may add execute, dispatch, apply, merge, commit, push, auto-fix, or
provider-race controls under this ADR.

### 10. Explicit non-goals

ADR-0006 does NOT authorize:

- Bridge-governed parallel slots.
- Provider-native parallel team execution controlled by cli-bridge.
- Worktree, branch, or shared-workspace isolation.
- Workspace-write auto-apply.
- Auto-commit, auto-push, auto-merge, PR creation, or merge queue.
- Scheduler, queue, daemon, background dispatch, or autonomous loops.
- ArbiterModel, ReplannerModel, SummarizerModel, bounded self-iteration, or
  model-driven conflict resolution.
- WorkBuddy executor promotion.
- Shell, exec, run, or command endpoints.
- Any bypass of ADR-0003 plan approval, step ceiling, per-step gate, audit, or
  failure-stop behavior.

### 11. ADR-0003 / ADR-0004 / ADR-0005 invariants

| Invariant | ADR-0006 position |
|---|---|
| Plan approval required before execution | Unchanged. |
| TeamSpec approval required before slot advance | Unchanged. |
| Sequential slot advance | Unchanged; concurrency remains 1. |
| Step ceiling hard 10 | Unchanged. |
| Per-step gate for state-changing steps | Unchanged. |
| Patch-only AgentTeam boundary | Unchanged. |
| Conflict reports read-only | Unchanged. |
| Model roles advisory-only | Unchanged for PlannerModel and CriticModel. |
| WorkBuddy non-executing | Unchanged. |
| No shell/exec/run/command endpoint | Unchanged. |
| No auto-apply/commit/push/merge | Unchanged. |

## Risk Acceptance

- **Provider identity confusion**: users may assume provider labels grant
  authority. Mitigation: policy uses provider id, endpoint id, and capability
  metadata only.
- **Capability asymmetry**: providers may expose different feature sets.
  Mitigation: shared capability schema and fail-closed unknowns.
- **Partial artifacts**: provider output may be incomplete or unstructured.
  Mitigation: only normalized, redacted, schema-valid artifacts participate in
  conflict reports and derived observability.
- **Conflict interpretation**: multi-provider artifacts may disagree.
  Mitigation: conflict reports remain read-only and require human review.
- **Scope creep toward parallelism**: multi-provider may be mistaken for
  multi-concurrent. Mitigation: hard `maxConcurrentBridgeSlots=1`,
  `bridgeGovernedParallelSlots=false`, and explicit rejection of parallel modes.

## Consequences

Accepted consequences:

- v2.4b may implement multi-provider AgentTeam identity, capability parity,
  sequential per-slot provider binding, artifact normalization, and read-only
  cross-provider conflict reporting.
- A separate implementation handoff must define exact file changes, request and
  response shapes, tests, verification commands, and closeout criteria.
- v2.5+ workspace-write, worktree, merge queue, and auto-merge remain deferred.

Rejected alternative, retained for decision history:

- AgentTeam remains single-provider.
- Multi-provider planning can continue as documentation only, or be replaced by
  a narrower design.

## Acceptance Conditions

The acceptance is conditional on the v2.4b implementation handoff satisfying all
of the following. A reviewer must confirm them at closeout:

1. Backward compatibility: existing single-provider TeamSpec payloads remain
   valid, and a slot with no provider defaults to the team-level provider.
   Covered by tests.
2. Hard-invariant tests: `maxConcurrentBridgeSlots=1`,
   `bridgeGovernedParallelSlots=false`, `isolationModes=['patch-only']`, and the
   existing "a slot is already executing" rejection still holds across providers
   (no parallelism).
3. Fail-closed tests: unknown capability fails closed; a provider failure during
   a slot stops the team and does not auto-start later slots; a partial artifact
   is recorded only when it passes redaction and SlotArtifact validation; a
   cross-provider conflict report is read-only (no winner selection, no apply).
4. Audit correlation includes `providerId`, `endpointId`, and `bridgeRunId`, and
   contains no raw provider output or API keys.
5. `canExecute=true` capability metadata remains inert: v2.4b adds no bridge
   dispatch/execution path that acts on it. New fields only extend existing
   TeamSpec routes and MUST NOT become a mutation or execution endpoint.
