# RP-WEB-AUTO-D - Release E2E Harness Hardening

Status: READY-FOR-EX

Date: 2026-06-19

## Review Basis

The Web automation stages are accepted:

- `docs/reviews/REVIEW-WEB-AUTO-A.md` is `PASS`.
- `docs/reviews/REVIEW-WEB-AUTO-B.md` is `PASS`.
- `docs/reviews/REVIEW-WEB-AUTO-C.md` is `PASS`.

Stage D is not a product-capability expansion. It exists because Stage B/C real
Chrome-for-Testing evidence was generated by one-off CDP scripts. That is enough
for the prior review, but too fragile for release repeatability.

## Goal

Create a repeatable local release harness for ChatGPT Web automation evidence.

The harness must:

- build the unpacked extension;
- launch or connect to a controlled Chrome-for-Testing profile;
- require an already logged-in ChatGPT account, and fail clearly when not
  logged in;
- start the local server on an available loopback port;
- inject the pairing token into extension session storage without printing it;
- run the accepted Stage B one-round and Stage C two-round hard-stop scenarios;
- write sanitized JSON evidence and screenshots under `output/playwright/`;
- close the local server and any harness-owned browser process on completion;
- document exact prerequisites and failure modes.

## Product Boundary

The harness is release evidence tooling only.

It must not change browser-extension behavior, server loop policy, ChatGPT DOM
submission logic, endpoint routing, or authorization semantics.

The browser extension remains the same Stage B one-round transport. Stage C loop
policy remains server-owned.

## Required Harness Behavior

### Inputs

The harness should accept explicit CLI options:

```text
--scenario stage-b-one-round | stage-c-two-rounds | all
--profile-dir <path>
--connect-cdp <url>                  optional alternative to --profile-dir; browser must already have extension loaded
--chrome-path <path>                 optional, default to Playwright CFT cache when discoverable
--remote-debugging-port <port>       optional, default available local port
--base-port <port>                   optional, default available local port
--output-dir <path>                  optional, default output/playwright/web-auto-release
--keep-browser                       optional, for manual debugging only
```

It may also support environment variables, but CLI options take precedence.

### Preconditions

The harness must fail before creating outbound work when:

- the extension dist cannot be built;
- Chrome-for-Testing cannot be launched or connected;
- the profile is not logged in to ChatGPT;
- the extension id cannot be discovered;
- the panel cannot reach `已连接` after token injection;
- ChatGPT is already streaming or the page is not in a stable state;
- another local server is already bound to the requested port.

### Evidence Contract

For each scenario, write:

```text
<output-dir>/<timestamp>-<scenario>.json
<output-dir>/<timestamp>-<scenario>.png
```

The JSON must include:

- scenario name;
- timestamp;
- git commit if available, and dirty-tree flag;
- extension id;
- Chrome version;
- server port;
- prompt ids and loop id, when applicable;
- outbound evidence state sequence;
- inbound marker checks;
- hard-stop result for Stage C;
- screenshot path;
- sanitized failure reason on failure.

The JSON must not include:

- pairing token;
- ChatGPT cookies;
- browser profile path contents beyond the path string supplied by the operator;
- full assistant reply text, except deterministic marker strings needed for
  acceptance.

### Scenarios

#### Stage B: one-round automatic return

Create one outbound prompt with a unique marker. Acceptance requires:

```text
queued -> claimed -> filled-and-acknowledged -> waiting-manual-send -> submitted -> responding -> response-ready -> returned
```

The inbound queue for the server-resolved endpoint must contain the marker.

#### Stage C: two rounds plus hard stop

Create a loop with `maxRounds = 2`. Acceptance requires:

- round 1 outbound reaches `returned`;
- round 1 inbound marker is present;
- advancing creates exactly one round 2 outbound;
- round 2 outbound reaches `returned`;
- round 2 inbound marker is present;
- the next advance returns `done` with `max-rounds-reached`;
- no third outbound is created;
- the ChatGPT page does not contain the sentinel third-round prompt marker.

## EX-WEB-AUTO-D Prompt

Implement release E2E harness hardening only.

Allowed files:

- `scripts/*web-auto*`
- `scripts/*e2e*`
- `package.json`
- `docs/runbooks/*web-auto*`
- `docs/planning/RP-WEB-AUTO-D.md`
- `docs/reviews/REVIEW-WEB-AUTO-D.md`
- focused tests for harness argument parsing, evidence redaction, port cleanup,
  and failure classification

Forbidden scope:

- modifying Stage A/B/C runtime behavior;
- changing ChatGPT DOM selectors or submission code;
- changing loop policy, outbound status transitions, inbound routing, or
  extension pairing behavior;
- adding MCP, shell, terminal control, workspace write, Git commit/push/merge,
  PR creation, deployment, or automatic approval;
- persisting pairing tokens or browser cookies;
- making the harness depend on a specific user profile path checked into source;
- treating unauthenticated ChatGPT as passable evidence.

Implementation requirements:

1. Extract the successful Stage B/C CFT flow into a reusable Node script.
2. Use structured helper functions instead of one monolithic eval script.
3. Discover or report the extension id after loading `apps/extension/dist`.
4. Start the local server through existing `startLocalServer` APIs.
5. Always close the server and harness-owned browser unless `--keep-browser` is
   explicitly set.
6. Write sanitized JSON evidence and screenshots.
7. Classify common failures: not logged in, extension missing, panel unpaired,
   ChatGPT timeout, outbound failed, inbound missing, hard stop failed, cleanup
   failed.
8. Add a runbook with prerequisites, exact commands, expected outputs, and
   troubleshooting.
9. Keep the default command non-destructive and local-only.

Required verification:

- `npm run typecheck`
- focused harness tests
- `npm run build-extension`
- `npm test`
- dry-run/failure-path check that does not require ChatGPT login
- one logged-in Chrome-for-Testing run:

  ```bash
  node --experimental-strip-types scripts/web-auto-release-e2e.ts --scenario all --profile-dir <logged-in-cft-profile>
  ```

- boundary scan:

  ```bash
  rg -n "requestSubmit|KeyboardEvent|\\.submit\\(|localStorage|document\\.cookie|CodexManaged|MockAgent|/bridge/run|/bridge/shell|/bridge/exec" apps/extension/src apps/extension/dist scripts
  ```

## REVIEW-WEB-AUTO-D Gate

Review must inspect:

- real diff and call chains;
- harness cleanup behavior;
- evidence redaction;
- failure classification;
- runbook accuracy;
- generated JSON and screenshot evidence from a real logged-in CFT run;
- historical boundary checklist:
  - no endpoint id in extension-owned active relay session;
  - no extension routing decision;
  - no browser secret reads;
  - no Enter-key, `requestSubmit`, or form `.submit()` fallback;
  - no shell/run/workspace/Git/MCP authority;
  - no automatic replay after uncertain submission.

Stage D passes only if the release harness can reproduce Stage B and Stage C
evidence from a logged-in Chrome-for-Testing profile and leaves no local server
or harness-owned browser process behind by default.
